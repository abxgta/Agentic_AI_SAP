Chapter One

Background — why this exists, and when to open it

Clients running SAP ECC and launching a transformation to SAP S/4HANA face a familiar temptation: customize the core to chase every use case. The stronger path — and the one this navigator supports — is to keep the S/4 core clean and simple, and push innovation to the edge through AI Insight Agents and AI Action Agents.

The transformation context

Dozens of use cases have already been collected from Order to Cash, Procure to Pay, Tax, and Treasury. Not all of them belong on an agentic roadmap — and treating every idea as an "AI use case" is itself a risk to manage.

Today's landscape

SAP ECC remains the transactional system of record. SAP BTP provides the extension and integration layer, SAP BDC (Business Data Cloud) harmonizes SAP data products, and Snowflake operates as the enterprise data lake that already blends SAP and non-SAP sources.

Where it's heading

SAP ECC is upgraded to SAP S/4HANA. The core is kept intentionally simple; differentiation happens at the edge, where Insight and Action Agents read from the data lake and, where appropriate, act back into SAP through governed, monitored interfaces.

When to open this navigator

Use it at the moment a business function hands over a raw idea, before any build effort is committed.
1

A use case is proposed

Business raises an idea from O2C, P2P, Tax, or Treasury — described in plain business language, not technology.

2

Fit is unclear

No one is yet sure whether it needs an agent, a dashboard, or a simple automation script.

3

Funding is being scoped

Leadership needs a defensible sprint and headcount estimate before it enters the roadmap.

4

A POD needs a backlog

A standing build POD needs its next quarter of sprints planned and staffed.

i

This is a qualification and estimation tool, not a build tool. It answers "should we, and how much will it take" — the actual data architecture, agent orchestration, and CI/CD pipeline work happen downstream, once a use case has cleared this front door.

Chapter Two

Overall approach — the agile operating model

Agents are not built one at a time in isolation. The operating model is domain-led and agile: lay the data foundation for a functional domain once, then run a repeatable cycle of selection, estimation, build, test, deploy, and hypercare for every use case inside it.

The delivery pipeline

Illustrated for one functional domain, e.g. Order to Cash. The same pipeline repeats for Procure to Pay, Tax, and Treasury, sharing common platform components where possible.
01

Data Foundation

Stand up the domain's data model on Snowflake, sourced from SAP ECC/S4 via BTP and BDC.

02

Use Case Selection

Run candidate use cases through the Agentic Use Case Analyzer.

03

Complexity & Effort

Classify complexity, then size sprints and quarters.

04

Build

The POD develops the agent, its tools, and guardrails inside a CI/CD pipeline.

05

Test

Functional, reasoning, and safety testing, including human-in-the-loop review of escalations.

06

Deploy

Controlled release into the target business process with defined rollback paths.

07

Hypercare

Elevated monitoring and rapid-response support immediately after go-live.

08

Continuous Monitoring

Ongoing performance, drift, and value tracking feeds the next backlog cycle.

↻

This is a loop, not a line. Continuous monitoring surfaces new candidate use cases and tuning opportunities, which re-enter Step 2 for the next sprint cycle — the core mechanism behind continuous learning and optimization.

Why domain-first, not agent-first

Building the data foundation once per domain avoids re-plumbing the same tables for every new agent, and keeps the S/4 core clean by pushing integration logic to the edge.

Reusable data products

Domain data is packaged once, with clear ownership and quality standards, so every agent in that domain can draw on trustworthy, governed data rather than building bespoke pipelines.

Shared orchestration layer

Agents, tools, and SAP systems connect through a common orchestration and guardrail layer, keeping the core simple even as edge intelligence grows.

Compounding velocity

Each subsequent agent in the same domain is faster to build than the last, because foundation, guardrails, and monitoring are already in place.

Chapter Three

Use case selection — qualifying for agentic AI

Before any complexity or effort conversation, every candidate must clear a single question: is this actually a job for an agent? Two agent archetypes exist on the "yes" side of that question — and a large, healthy share of use cases will land on the "no" side.

Two agent archetypes

Both extend generative AI beyond passive content generation — the distinction is whether the agent stops at a recommendation, or carries it through to action inside SAP and adjacent systems.
Decision Support

AI Insight Agent

Perceives data, reasons over it, and delivers a recommendation, explanation, or answer — a human remains the one who acts.

  • Retrieves, correlates, and explains data across systems
  • Produces recommendations, forecasts, or narratives
  • Escalates ambiguity to a human rather than resolving it
  • Does not directly change a system of record
Example: a Treasury cash-position chatbot that explains a liquidity shortfall and recommends a funding option for a human to approve.
Autonomous Execution

AI Action Agent

Plans, decides, and executes multi-step actions across systems toward a defined goal — with humans supervising rather than performing the task.

  • Breaks a goal into subtasks and sequences them
  • Calls tools and APIs to change records or trigger workflows
  • Operates within bounded autonomy and escalates high-impact decisions
  • Learns from outcomes to improve the next cycle
Example: a Procure-to-Pay exception agent that matches, resolves, and posts routine invoice discrepancies without manual intervention.

Chapter Four

Identify the complexity of the agent

Once a use case is confirmed as agentic, it needs a common complexity language before anyone can size the effort. Three tiers — High, Medium, Low — describe how much autonomy, orchestration, and governance the agent actually requires.

High

Tier 3
  • Operates across multiple complex domains simultaneously
  • High autonomy: goal-driven orchestration with planning, multi-step reasoning, and dynamic decisions
  • Orchestrates multiple sub-agents and components
  • Handles complex technical requirements, e.g. non-standard or high-volume APIs
  • Requires continuous learning and user-specific adaptation
  • Demands high governance, monitoring, and risk controls
POD velocity1 per sprint

Medium

Tier 2
  • Operates across multiple related domains or tasks
  • Moderate autonomy with conditional branching, data retrieval, validation, and minor reasoning
  • Multiple system integrations using standard APIs
  • Context-aware with some personalization
  • Requires moderate monitoring and exception management
POD velocity2 per sprint

Low

Tier 1
  • Operates in a single domain with limited scope
  • Well-defined, rule-based tasks with minimal autonomy and few integrations
  • Minimal decision-making; mostly reactive
  • Limited need for personalization or contextual understanding
  • Easy to test and deploy with standard guardrails
POD velocity3 per sprint

How to apply the tiers

Score the use case against each bullet within a tier; the tier where most descriptions genuinely fit — not the most ambitious tier — is the correct classification.

Common misclassification

Teams often over-classify a use case as High complexity because it feels strategically important. Complexity should reflect technical and orchestration difficulty, not business priority — a High-priority use case can still be Low complexity.

Why the tier matters downstream

Complexity directly drives POD velocity on Page 5 — a High complexity agent consumes an entire sprint's capacity, while three Low complexity agents can share one.

Chapter Six

Build POD — team and duration

Effort estimates only become real when a POD is staffed against them. Every build POD follows the same core structure, working in six two-week sprints per quarter, capped at 45 hours per person per week.

Core POD roles

Roles scale with domain scope; the roster below is the minimum standing team for one active functional domain.
◇

Scrum Master

Runs the sprint cadence, removes blockers, and keeps the POD accountable to the quarter's committed backlog.

≤ 45 hrs / wk
▦

Data Architect (Snowflake)

Owns the domain data model, ingestion patterns, and data product standards feeding every agent.

≤ 45 hrs / wk
⚙

Data Engineer

Builds and maintains the pipelines that move data from SAP ECC/S4, BTP, and BDC into the lake.

≤ 45 hrs / wk
⌘

SAP Functional Expert

Validates business process logic and ensures actions taken by agents are correct inside SAP.

≤ 45 hrs / wk
◈

AI Agent Developer

Designs, builds, and hardens the agent — prompts, tools, orchestration, guardrails, and the CI/CD pipeline.

≤ 45 hrs / wk
+

Extend as needed

Add specialist roles — security, change management, forward-deployed engineer — for higher-complexity domains.
